Correction de l'exercice : Dépendances fonctionnelles et normalisation

1. Détermination de la clé de la relation R

Pour trouver la clé de la relation R, on identifie les attributs qui n'apparaissent jamais à droite (la partie membre droit) d'une dépendance fonctionnelle (DF). Ces attributs doivent obligatoirement faire partie de toute clé candidate.

En analysant vos dépendances fonctionnelles :

  • NumEmploye n'est jamais déterminé par un autre attribut.
  • NumProjet n'est jamais déterminé par un autre attribut.

Calculons la fermeture de l'ensemble contenant ces deux attributs, notée {NumEmploye, NumProjet}+ :

  • On commence avec : {NumEmploye, NumProjet}
  • En utilisant NumEmploye → NomEmploye, CodeService, on obtient :
    {NumEmploye, NumProjet, NomEmploye, CodeService}
  • En utilisant CodeService → NomService, on obtient :
    {NumEmploye, NumProjet, NomEmploye, CodeService, NomService}
  • En utilisant NumProjet → NomProjet, BudgetProjet, on obtient :
    {NumEmploye, NumProjet, NomEmploye, CodeService, NomService, NomProjet, BudgetProjet}

Puisque la fermeture contient l'intégralité des attributs de la relation R et qu'aucun sous-ensemble plus petit ne peut le faire, cet ensemble forme la clé primaire.

Clé de la relation R : (NumEmploye, NumProjet) (C'est une clé composée).

2. Décomposition de la relation R en troisième forme normale (3FN)

La relation originale R n'est pas en 2FN car il y a des dépendances partielles (des attributs non clés dépendent d'une partie seulement de la clé composée). Elle viole aussi la 3FN à cause de la dépendance transitive CodeService → NomService.

Pour la décomposer en 3FN tout en préservant les dépendances fonctionnelles et la clé, on crée une relation distincte pour chaque groupe de dépendances logiques, plus une relation pour l'association globale :

  • R1 (Employés) : (NumEmploye, NomEmploye, CodeService)
    Provenant de : NumEmploye → NomEmploye, CodeService

  • R2 (Services) : (CodeService, NomService)
    Provenant de : CodeService → NomService

  • R3 (Projets) : (NumProjet, NomProjet, BudgetProjet)
    Provenant de : NumProjet → NomProjet, BudgetProjet

  • R4 (Affectations) : (NumEmploye, NumProjet)
    Cette relation est indispensable pour maintenir le lien initial entre les employés et les projets sur lesquels ils travaillent (préservation de la clé d'origine).
Résultat final : La décomposition produit 4 relations saines en 3FN : R1, R2, R3 et R4.

Exercice IV - Partie B : Requêtes SQL (9 pts)

Voici les scripts de création des tables manquantes ainsi que les requêtes de manipulation de données (DML) et de sélection demandées.

1 Création des tables (DDL)

Définition de la structure des tables Produit, Achete et Note avec contraintes d'intégrité (clés primaires et clés étrangères).

CREATE TABLE Produit (
    Code VARCHAR(50) PRIMARY KEY,
    Libellé VARCHAR(100),
    Marque VARCHAR(50),
    Type VARCHAR(50),
    Prix DECIMAL(10, 2),
    DateFabrication DATE
);

CREATE TABLE Achete (
    idClient INT,
    codeProduit VARCHAR(50),
    numMagasin INT,
    DateAchat DATE,
    PRIMARY KEY (idClient, codeProduit, numMagasin),
    FOREIGN KEY (idClient) REFERENCES Client(ID),
    FOREIGN KEY (codeProduit) REFERENCES Produit(Code),
    FOREIGN KEY (numMagasin) REFERENCES Magasin(Num)
);

CREATE TABLE Note (
    idClient INT,
    codeProduit VARCHAR(50),
    Note INT,
    PRIMARY KEY (idClient, codeProduit),
    FOREIGN KEY (idClient) REFERENCES Client(ID),
    FOREIGN KEY (codeProduit) REFERENCES Produit(Code)
);
2 Mise à jour des prix

Augmentation de 10% du prix de tous les produits de la marque 'Samsung'.

UPDATE Produit
SET Prix = Prix * 1.10
WHERE Marque = 'Samsung';
3 Suppression des produits invendus

Suppression des produits dont le code n'apparaît dans aucune transaction de la table Achete.

DELETE FROM Produit
WHERE Code NOT IN (SELECT DISTINCT codeProduit FROM Achete);
4 Sélection des marques de smartphones de 2023

Liste des marques distinctes de smartphones fabriqués entre le 1er janvier et le 31 décembre 2023.

SELECT DISTINCT Marque
FROM Produit
WHERE Type = 'Smartphone' 
  AND DateFabrication BETWEEN '2023-01-01' AND '2023-12-31';
5 Clients de Lyon ayant acheté un iPhone 15

Sélection des informations des clients résidant à Lyon ayant effectué l'achat d'un produit nommé 'iPhone 15'.

SELECT DISTINCT c.*
FROM Client c
JOIN Achete a ON c.ID = a.idClient
JOIN Produit p ON a.codeProduit = p.Code
WHERE c.Ville = 'Lyon' AND p.Libellé = 'iPhone 15';
6 Statistiques des produits bien notés

Affichage de la marque, du code et du libellé des produits, ainsi que le nombre de clients uniques ayant attribué une note strictement supérieure à 4.

SELECT p.Marque, p.Code, p.Libellé, COUNT(DISTINCT n.idClient) AS NbClients
FROM Produit p
JOIN Note n ON p.Code = n.codeProduit
WHERE n.Note > 4
GROUP BY p.Marque, p.Code, p.Libellé;
